## Overview / problem statement

Every prior slice (read, install/drift, working-copy, versioning, overlays, metadata, lifecycle, reimport, search, bootstrap) kept the bridge **network-free and secrets-free**: `crates/prompt-library-bridge/Cargo.toml` deliberately does not link `prompt-library-secrets`, so a `SecretStore` is unconstructible at compile time (`Cargo.toml:12-17`), and `main.rs`'s module doc asserts "no network, no secrets on ANY path" (`main.rs:22-24`).

Slice 8 breaks that invariant **exactly once**. It ports the full mutating git story:

1. **Configure remote** — validate + persist a GitHub HTTPS URL.
2. **PAT storage** — set/delete a GitHub Personal Access Token in a `SecretStore`; never return it across the wire (only the redacted form).
3. **Remote status** — remote URL + redacted PAT + unpushed-commit count.
4. **Push** — `git push` (with first-push `-u origin <branch>` detection), gated by a **secret-scan that must block on a finding** (the UI runs the scan first, surfaces findings, and only calls push after the user reviews).
5. **Pull** — `git pull --rebase`, which can pause mid-rebase awaiting conflict resolution. The conflict flow (`is_pull_paused` → `list_pull_conflicts` → `read_conflict_blob` → `resolve_conflict` → `continue_pull` / `abort_pull`) is **stateful across multiple HTTP requests** — the only such flow in the consolidation.

The whole reference command surface is `commands.rs:1594-2052`; the crate functions it calls (`git/src/{git_ops,conflict,push_gate,secret_scan,askpass}.rs`, `secrets/src/keychain.rs`) are **already imported and unit-tested** in this repo. This slice is bridge wiring + a Cargo dependency add + TS routes/models with a hard secret-redaction discipline + a git-sync UI panel.

### Load-bearing risks discovered at source (decisions this plan must settle)

1. **The Keychain ACL is bound to the binary's signature/path and re-prompts whenever the binary hash changes** (`keychain.rs:14-19` — verbatim: "The prompt re-appears whenever the binary hash changes (e.g. after `cargo build`)…No automated test"). Under a **process-per-request bridge that the dashboard rebuilds on every `cargo build`**, this means every dev rebuild re-prompts for keychain consent, *and a one-shot bridge process must succeed non-interactively for set/get/delete to work headless*. This is the **single biggest unknown in the slice** and drives the PAT-storage decision below.
2. **`get_pat` is called on every `push_now`/`pull_now`/`get_remote_status`** — i.e. a fresh keychain read per bridge invocation (the long-lived Tauri process read it once). Must verify a one-shot process can read the item it wrote in a *prior* one-shot process without re-consent (likely yes once "Always Allow" is granted to a *stable installed* binary; likely a re-prompt storm in *dev*).
3. **Askpass needs a persisted state-dir.** `init_askpass_script(state_dir)` must have been run (writing `git-askpass.sh`) before `push`/`pull`, and `state.askpass_path` must point at it. The bridge has no launch hook. **Decision: the bridge writes/re-writes the askpass script into a deterministic `DATA_DIR/askpass/` (TS-injected, like every other dashboard-owned path) at the top of each push/pull command** — `init_askpass_script` is idempotent + atomic + cheap (a single static-body write), so doing it per-invocation is correct and removes the launch-hook requirement.
4. **Mid-rebase state lives in `.git`, not in the process** (`conflict.rs:63` — `is_rebase_in_progress` reads `.git/rebase-merge`|`rebase-apply`). **This is the saving grace** for the stateful conflict flow: a one-shot bridge can `pull_now` (pause), exit, and a *later* one-shot `list_pull_conflicts`/`resolve_conflict`/`continue_pull` reads the same on-disk rebase state. No process-resident session needed. **The TS write mutex must serialize the whole push/pull family** so two concurrent requests don't race the rebase.
5. **Secret leakage is the dedicated-security-agent's whole job.** The PAT must never reach: (a) a log line (`errorResult` logs `detail` server-side — the PAT must never be *in* a detail); (b) a client response body; (c) git argv (askpass keeps it in env — already handled in-crate); (d) the bridge's stderr (which TS surfaces in transport errors). The wire form is **always `redact_pat`**. A tripwire test asserts no raw PAT in any log/error body.

## Decisions (settled here)

- **D1 — Remote URL persistence: dashboard config, not core `Settings`.** `configure_remote` validates in the bridge and *returns* the normalized URL; the **TS route writes it to `config/library.yaml`** (`remote_url:` key) and `loadLibraryConfig` reads it back into `LibraryConfig.remoteUrl`. Rationale: the dashboard already owns `config/library.yaml` for `library_path` and explicitly does NOT use core `Settings`; threading a second persistence mechanism through the bridge would fork the config story. The bridge stays config-file-free (consistent with how it stays clock-free — TS supplies `created_at`). This is a deliberate divergence from the reference's `state.set_remote_url`.
- **D2 — PAT storage: `KeychainStore` in production, `InMemoryStore` injectable for tests; bridge constructs it per-invocation.** Match the reference's Keychain choice (it is the right place for a secret on macOS — not a plaintext config file). The bridge gains a `--secret-store` posture: real `KeychainStore` by default, swappable to `InMemoryStore` under a test/env flag so `cargo test` never blocks on a consent prompt. **Open the dev-rebuild re-prompt as a known friction** (documented, "Always Allow", matches the standalone app's own dev experience) — do NOT invent a config-dir secret to dodge it; that would weaken the security posture for a dev-only annoyance. *(See Open Question 1 — the headless one-shot keychain read is the one thing to verify on the real machine before committing.)*
- **D3 — Askpass per-invocation into `DATA_DIR/askpass/git-askpass.sh`** (TS-injected `askpass_dir`, default `DATA_DIR/askpass`). `init_askpass_script` runs at the top of `push_now`/`pull_now` (idempotent). No launch hook. Path injected the same way `installs_path`/`session_path` are (D7 boundary: never from an HTTP body).
- **D4 — Push gate BLOCKS, never auto-bypasses.** `scan_before_push` is its own command + route; the UI calls it, renders every finding (`path:line`, kind, matched bytes verbatim — the user must see what tripped it), and only enables "Push anyway" after review. `push_now` does NOT re-run the gate (matches reference). The gate-then-push handshake is a UI sequencing contract, asserted in component tests.
- **D5 — The whole push/pull/conflict family serializes under `withWriteLock`.** Even the read-shaped ones (`scan_before_push`, `count_unpushed_commits`, `list_pull_conflicts`, `read_conflict_blob`, `is_pull_paused`) — because the rebase state in `.git` is shared mutable state and a `resolve`/`continue` racing a `list` would read a torn view. The pure-status reads (`get_remote_status`) can skip the lock. **Network timeout > write timeout:** push/pull get a `NETWORK_TIMEOUT_MS` (settle the value in implementation — the reference uses a 2s *launch-hook* pull cap, but an interactive user-initiated push/pull should tolerate a slow-but-healthy network; propose 60s, longer than `WRITE_TIMEOUT`'s 30s, and bound it so a hung network can't wedge the write chain forever).
- **D6 — Secret redaction is a tested tripwire, not a convention.** A `library_bridge`/`library_routes` test plants a known PAT and asserts it appears in **no** log line and **no** response body across set/get/status/push-error paths. The only PAT-display form on any surface is `redact_pat`.
- **D7 — New error codes + HTTP mapping.** Add to `statusForCode`: `empty_pat`→422, `no_pat_stored`→409 (a precondition: configure a PAT first), `remote_not_configured`→409, `invalid_remote_url`→422, `git_failed`→502, `git_timed_out`→502 (or a dedicated 504-shaped code — settle in impl; the existing map has no 504, so 502 keeps the contract simple), `pull_conflict`→**200 as DATA** (it's a routable outcome the UI renders a resolve-banner for, like `InstallSummary`/`ReimportResult` — NOT an error envelope). `secret_scan_findings`→200 as data (the findings list is the payload). Mirror the read/install slices: outcomes the UI routes on ride the `ok` envelope; only genuine faults are error envelopes.

## Acceptance criteria
- The full reference git-sync surface (`commands.rs:1594-2030`, minus the native `reveal_conflict_path`) is ported as bridge arms + routes + UI; `reveal_conflict_path` is replaced by a copy-absolute-path affordance (Slice 10c decision log).
- The bridge links `prompt-library-secrets` and constructs a `SecretStore` **only** on the git-sync path; every other command path constructs none (tested assertion preserves the network-free/secrets-free posture elsewhere).
- The raw PAT appears in **no** log line, **no** response body, and **no** git argv on any path; the only display form is `redact_pat` (tripwire test).
- `scan_before_push` blocks push on a finding: push proceeds only after an explicit user confirm; the findings are surfaced with `path:line`+kind+matched.
- The stateful pull-conflict flow works across separate one-shot bridge invocations (rebase state read from `.git`), serialized under the write mutex.
- A failed configure/push/pull/scan leaves `/api/summary`, `/api/agents`, `/healthz`, and doctor at 200 (route-local failure, tested).
- The remote URL persists to `config/library.yaml` without clobbering `library_path`; `loadLibraryConfig` reads it back.
- Every new UI state is CVD-safe (label+glyph+Okabe-Ito tone, no bare red/green).
- Gates green: `cargo test --workspace` (incl. the bare-remote fixture + scan-blocks-secret + PAT-never-returned), `bun test scripts` (incl. the redaction tripwire + stateful-flow mapping), `*.svelte.test.ts` (push-gate confirm, conflict resolver, PAT never echoed), `bun run check` / svelte-check / clippy clean.

## Dependencies and risks
- **Depends on:** every prior slice (the bridge dispatch, `withWriteLock`, `statusForCode`, `commit_change`, the `Library.svelte` Git section, the `library_config` seam). Nothing new must be imported — `crates/{git,secrets}` are present.
- **Top risks (ranked):**
  1. **Keychain headless one-shot read** (Phase 0 checkpoint) — if it re-prompts per invocation, D2 needs rework. STOP-and-ask.
  2. **Secret leakage** — the dedicated-security-agent's focus; the D6 tripwire is the guard.
  3. **Network timeout vs. write-mutex wedge** — a hung push holding the write chain; bound `NETWORK_TIMEOUT_MS` and verify the chain recovers on timeout.
  4. **Config-file write** — first route to mutate `config/library.yaml`; must be atomic + key-preserving.
  5. **Rebase-state correctness** across one-shot invocations — mitigated by `.git`-resident state + the write mutex, but the conflict loop is the most stateful UI in the consolidation; test the StillConflicted re-loop.

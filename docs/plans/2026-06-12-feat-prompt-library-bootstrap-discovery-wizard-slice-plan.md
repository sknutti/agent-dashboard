## Overview / problem statement

The Library route today assumes a library that already has primitives in it. The **first-run gap**: a user with existing `~/.claude/skills`, `~/.pi/agent/skills`, `~/.codex/prompts` etc. has no path to pull that scattered, hand-authored content *into* the library in one pass. The bootstrap discovery wizard closes it: **scan** the user's install roots тЖТ **classify** each candidate against the library (already-imported / drifted / new / needs-review) тЖТ let the user **review and deselect** тЖТ **execute** (create the new ones at v1, reimport the drifted ones as vN+1) тЖТ commit once, with a **resumable session** so a crash or a mid-run skip can be retried without double-creating.

The entire engine is **already imported and exhaustively tested in `crates/core`** тАФ `bootstrap_scan` (the orchestrated scan + 3-stage progress callback), `derive_plan` (pure `CrossReferenced` тЖТ `BootstrapPlan`), `bootstrap_execute` (the checkpointed orchestrator with backup + resume + per-item session save), and `BootstrapSession` (atomic save/load/clear). This slice is **bridge wiring + two new config paths + TS routes/models + a multi-step wizard UI** тАФ no new core logic. It is the most stateful UI in the consolidation after git conflict resolution, but the statefulness lives in `core` (the session file) and the `BootstrapPlan` the wizard holds between scan and execute тАФ not in the bridge process.

### The "hard part" is smaller than the roadmap assumed

The roadmap (line 203) flags progress-streaming-over-a-one-shot-bridge as the load-bearing risk and leans toward session-file polling. **Reading the source revises that down:** `ScanProgress` (`bootstrap_scan.rs:18-26`) is a closed 2-variant enum that emits **exactly three coarse `Stage` events** ("Scanning install rootsтАж", "Deduplicating candidatesтАж", "Cross-referencing library stateтАж") followed by **one `Done`** carrying the banner summary. It is **not** a per-file stream тАФ it is a fixed 3-phase progress bar over a bounded walk of a handful of `~/.claude`/`.pi`/`.codex` roots (`scanner.rs` `SCAN_MATRIX`). At ~117 primitives the search slice benched a full library walk at ~50-80ms; the bootstrap scan is the same order of magnitude. **Decision (Open Q1, settled below): drop the progress channel entirely** тАФ run the scan as a single requestтЖТsingle-response bridge call under a generous read timeout, and render the three stages as a client-side indeterminate progress label, not a server-streamed one. The `Channel<ScanProgress>` is a Tauri affordance with no web equivalent and no payoff here; we pass a **no-op `on_progress` closure** to `bootstrap_scan` (exactly as core's own tests do тАФ `bootstrap_scan.rs:121` `|_| {}`). This is consistent with the dashboard's poll-don't-push posture (ADR-0008) and the `subscribe_drift`-deferred precedent, but goes further: with 3 events and a sub-second-to-low-seconds scan, we don't even need to poll.

- `crates/core/src/bootstrap.rs:59` тАФ `derive_plan(cr: CrossReferenced) -> BootstrapPlan`. **Pure.** Drops `AlreadyImported` and `NeedsManualReview`; maps `New` тЖТ `CreateAction`, `Drifted` тЖТ `ReimportAction`. `BootstrapPlan { creates: Vec<CreateAction>, reimports: Vec<ReimportAction> }` (`:29`).
- `crates/core/src/bootstrap.rs:149` тАФ `bootstrap_execute(req: BootstrapExecuteRequest) -> Result<BootstrapExecuteSummary, Error>`. The checkpointed orchestrator:
  - Takes-or-makes a `BootstrapSession` (fresh `new(timestamp)` or the passed `resume`), sets `excluded_ids`.
  - Writes the **source-dir tarball backup** once (`create_source_backup`, `source_backup.rs:30`) unless `session.backup_taken`; persists the session after the backup.
  - `session.filter_remaining(plan)` drops already-completed items, then runs each create (`execute_one_create`: writes metadata + working + v1 snapshot + install records, saves `installs.json`, `record_create`, saves session) and each reimport (`execute_one_reimport`: synthesizes an install record then routes through `reimport_install_as_version`, `record_reimport` on success, restores `installs.json` on a skip).
  - Per-item `session.save` after every success тЖТ a crash leaves a resumable checkpoint.
  - Clears the session iff `summary.skipped == 0`; otherwise re-saves it so the wizard finds "N items need review" on relaunch.
  - `BootstrapExecuteSummary { backup_path: Option<Utf8PathBuf>, created, reimported, skipped, skipped_items: Vec<BootstrapSkippedItem> }` (`:117`). `BootstrapSkippedItem { kind, name, source_target, reason }` with `reason тИИ { WorkingCopyDirty, InstallMissing }` (`:132-145`).
- `crates/core/src/bootstrap_session.rs` тАФ `BootstrapSession { format_version, started_at, backup_taken, excluded_ids, completed }` + `load(path) -> Result<Option<Self>>` (None when absent; `InstallsParse` error on garbage тЖТ caller treats as non-resumable), `clear(path)` (no-op when absent), atomic `save`. `FORMAT_VERSION = 2`. **`excluded_ids: Vec<String>`** are the action ids the user unchecked in review, persisted so a resume preserves the choices.
- `crates/core/src/cross_reference.rs` тАФ `CrossReferenced { groups: Vec<ClassifiedGroup>, needs_manual_review, symlinked, unclassified }`; `Classification` (`:52`) = `New { content }` / `AlreadyImported` / `Drifted { content }`; `CrossReferenceSummary { new, already_imported, drifted, needs_manual_review }` (`:66`, the wizard's banner). `ClassifiedGroup { kind, name, classification }`.
- All of the above are `Serialize + Deserialize + specta::Type` тАФ they cross the bridge JSON boundary as-is. **`BootstrapPlan`, `BootstrapSession`, and the `excluded_ids` list round-trip TSтЖФbridge** (execute takes the plan + optional resume + excluded ids that the scan/derive step produced).

### Reference command bodies (the port target тАФ strip `AppHandle`/`State`/`Channel`)

`prompt-library/src-tauri/src/commands.rs`:

- `bootstrap_execute` (`:1517`): resolves `library`, `home`, `installs_path`, `session_path`, `backup_dir`, stamps `now_rfc3339_filesafe()`, runs `core_bootstrap_execute(BootstrapExecuteRequest{ plan, layout, install_paths, installs_file, session_path, backup_dir, home, timestamp, resume, excluded_ids })`, then `commit_change(&library, "bootstrap: created N, reimported M")`. **The `now` and the four paths are our seam:** the TS route supplies `created_at` (shape-checked like publish/reimport тАФ the bridge owns no clock) and the route injects `session_path`/`backup_dir` from config (never the body тАФ the D7 containment boundary).
- `read_bootstrap_session` (`:1564`): `bootstrap_session_path_for(app)` тЖТ `BootstrapSession::load`. Returns `Option<BootstrapSession>`.
- `clear_bootstrap_session` (`:1575`): `BootstrapSession::clear`.

Note the reference uses a **filesafe** timestamp (`now_rfc3339_filesafe`) because `timestamp` becomes the backup tarball filename (`<timestamp>.tar.gz`, `source_backup.rs:48`) **and** the `metadata.created_at` / version `created_at` written verbatim. A standard `toISOString()` contains `:` (illegal in some filename contexts, ugly on macOS). **Settle in Phase 1 (Open Q2):** the bridge sanitizes the TS-supplied RFC3339 timestamp into a filesafe form *for the backup filename only* (replace `:` тЖТ `-`), while passing the original to core for `created_at`. The reference collapses these into one filesafe string; verify whether `created_at` consumers care (version metadata stores it as a display string тАФ a filesafe variant is harmless but loses the canonical `:`). Default: **sanitize only the filename, keep canonical RFC3339 for `created_at`** тАФ a small divergence that keeps version timestamps canonical.

### Bridge (extend тАФ 4 dispatch arms, two new config-path args)

- Dispatch table `main.rs:111-198`. Add four arms grouped as a "bootstrap discovery slice" block (mirroring the lifecycle block's doc comment):
  - `"bootstrap_scan" => cmd_bootstrap_scan(args)` тАФ **sync** (the scan is pure `std::fs`, no commit, no ledger mutation; like `find_in_library`). Read-only against the library + home.
  - `"bootstrap_execute" => cmd_bootstrap_execute(args).await` тАФ **async** (it COMMITS the created/reimported version trees, like publish/reimport).
  - `"read_bootstrap_session" => cmd_read_bootstrap_session(args)` тАФ **sync**, read-only.
  - `"clear_bootstrap_session" => cmd_clear_bootstrap_session(args)` тАФ **sync** (removes the session file; no commit).
- **`derive_plan` is the open seam.** The reference's `bootstrap_scan` returns `CrossReferenced`; `bootstrap_execute` takes a `BootstrapPlan`. `derive_plan` (pure) sits between them. **Decision (Open Q3): run `derive_plan` inside `cmd_bootstrap_scan` and return both `CrossReferenced` (for the review UI's full classification + needs-review/symlinked/unclassified lists) and the derived `BootstrapPlan` (the executable subset) in one envelope** тАФ `{ cross_referenced, plan }`. This keeps the plan derivation server-side (the UI never re-implements the New/DriftedтЖТaction mapping) and lets the review step show the full picture while editing only the executable `excluded_ids`. The alternative (a separate `derive_plan` arm) adds a round-trip for a pure function; reject it.
- **`require_library`** (`main.rs:232` pattern) resolves the library root. **`install_context`** (`main.rs:1228`) resolves `(InstallPaths, installs_file_path)` from `home` + `installs_path` тАФ reused verbatim by execute.
- **Two new config-injected paths** the bridge must read from args (never the body): `session_path` and `backup_dir`. These follow the `installs_path` precedent (`install_context` reads `args["installs_path"]`, errors `installs_unconfigured` when empty). Add `parse_session_path` / `parse_backup_dir` (or fold into a `bootstrap_paths(args)` helper) with the same "missing тЖТ config fault, not user error" posture.
- **Commit seam:** `commit_change(&root, "bootstrap: created N, reimported M")` (`main.rs:1466`) тАФ the shared non-fatal `(committed, commit_error)` helper. Folded onto the execute response object exactly as reimport/publish do (`main.rs:935`). Commit runs **only when something was written** (`created + reimported > 0`); a scan-only or all-skipped/all-excluded run commits nothing (mirror reimport's "commit only on the outcome that wrote git-tracked bytes"). **Note:** the reference always commits; we gate on `created + reimported > 0` to match the dashboard's reimport precedent (no empty/misleading commits).
- **Error mapping (`map_core_error`, `main.rs:~1182`):** bootstrap's reachable `core::Error` variants are `Io` (тЖТ `library_unreadable`, mapped), `InstallsParse`/`InstallsSerialize` (the session/ledger тАФ confirm mapping; likely the catch-all 502, acceptable as a "dashboard's own state file is corrupt" fault), `InvalidVersionLabel` (the `next_version` auto-bump guard for a `-suffix` current label, `bootstrap.rs:382` тЖТ likely `library_invalid_version` 422, **verify it's mapped**), `NoCurrentVersionForInstall` (`bootstrap.rs:305` тАФ a Drifted reimport whose library primitive has no `current.txt`; **verify mapping**, likely catch-all). **No new error arm is expected**, but the two version/no-current variants are the ones to confirm тАФ promote to a typed code only if a reviewer wants them actionable.

## Acceptance criteria

- A user with existing `~/.claude`/`.pi`/`.codex` content can run the wizard: scan classifies every candidate (new/drifted/already-imported/needs-review) in one bridge call; review lets them deselect; execute creates the new ones at v1 and reimports the drifted ones as vN+1; the library is committed once ("bootstrap: created N, reimported M"). Verified end-to-end in a `cargo` bridge test and asserted in the UI.
- The scan runs as a **single requestтЖТsingle-response** call with **no progress channel** тАФ the three stages render as a client-side indeterminate label (Open Q1 settled: drop the Tauri `Channel`).
- `bootstrap_scan` returns both the full `CrossReferenced` and the derived `BootstrapPlan` in one envelope (Open Q3: `derive_plan` server-side, no extra round-trip).
- A **source-dir tarball backup** is written before any library write on a fresh run (skipped on resume), and its path is surfaced to the user.
- Execute commits the created/reimported version trees **non-fatally** (`{committed, commit_error}`, publish posture) **only** when `created + reimported > 0`; a scan-only / all-excluded run commits nothing.
- A partial run (skipped `WorkingCopyDirty`/`InstallMissing` items) leaves the **session on disk**; relaunch offers Resume/Discard, and Resume retries only the unresolved items without re-creating completed ones or re-writing the backup (the core resume contract).
- Execute takes `withWriteLock` (it mutates `installs.json`), unlike its publish sibling тАФ asserted by a concurrency tripwire. The four config paths (`home`, `installs_path`, `session_path`, `backup_dir`) are injected from config, never the HTTP body (D7) тАФ asserted.
- All execute outcomes (including `skipped_items`) ride HTTP 200 as **data**; only genuine core faults map to 422/404/502. The absent-session read is a 200 `null`, never a 404.
- CVD-safe cues throughout (Scott is red/green colorblind): classification and skip-reason states are label + glyph + Okabe-Ito tone, distinguishable with color stripped.
- Route-local failure preserved: a failed scan/execute leaves `/api/summary`, `/api/agents`, `/healthz`, doctor at 200.
- Secrets-free + network-free intact: no `prompt-library-secrets` link, no network call (the `Cargo.toml:12-14` comment untouched) тАФ bootstrap is local-fs only.

## Dependencies and risks

- **Depends on:** Slice L (lifecycle тАФ `scaffold_primitive` is the create path bootstrap's `execute_one_create` uses; the explorer affordance pattern; the unwired `forget`) and Slice 7 (reimport тАФ `reimport_install_as_version` is the reimport path bootstrap's `execute_one_reimport` routes through; the `withWriteLock`-on-ledger-mutating-route precedent; the partial-outcome-rides-200 posture). Both shipped.
- **Atomicity / partial-failure (the headline):** `bootstrap_execute` is a **bulk multi-step write** тАФ backup, then per-create (metadata + working + snapshot + install records + session save), then per-reimport (synthesize record + reimport + session save). It is **not atomic across items**, but it is **resumable**: each successful item is checkpointed to the session before the next, each file is written atomically (temp+rename), and a skip restores `installs.json`. A killed bridge mid-execute leaves a session whose `completed` reflects exactly the items that landed; the next run's `filter_remaining` skips them тАФ **no double-create**. The slice owes an explicit "not atomic across items, but checkpointed + resumable, never double-creates" statement + a kill-mid-execute test (the D3 pattern, extended to the bulk case тАФ the core already proves resume; the bridge test proves the envelope + commit gating survive it).
- **Scan timeout:** the scan walks the home tree once; at ~117 primitives it's ~50-80ms (the search-slice bench), but a pathological home (huge `.claude`) could be slower тАФ hence a dedicated `BOOTSTRAP_SCAN_TIMEOUT_MS` (30s) rather than the 10s read default. If a real home ever approaches that, revisit (an index or a bounded scan), but don't pre-optimize.
- **Resume/plan consistency:** the session matches completed items by `(kind, name, action)`; a re-scan that changed the candidate set between a partial run and a resume could mismatch. Mitigation: the wizard surfaces "the library changed since this session" when the fresh scan no longer aligns, and offers Start-over.
- **Filename-vs-`created_at` timestamp:** the backup filename must be filesafe; `created_at` should stay canonical RFC3339. Settle by sanitizing only the filename in the bridge (Open Q2).
- **`forget` scope creep:** the Reconcile view is the natural home for `forget`, but building it expands this slice. Defer it (Open Q4) and ship the wizard focused.

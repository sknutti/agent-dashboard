## Overview / problem statement

The Library detail pane shows a Primitive's metadata **read-only** today (`Library.svelte:676-677` display_name; `:897-914` allowed_targets / author / created_at / source_url). There is no way to edit it through the dashboard — an author must hand-edit `metadata.yaml`. This slice ports the reference's single `update_metadata` command and adds a metadata form that edits the **three editable fields** (`allowed_targets`, `display_name`, `author`) through the Rust core, with the same validation the reference enforces and the same comment-preserving splice already in-tree.

The slice's value is **surfacing core's existing validation as typed route codes + an editing form**, not reimplementing validation. Two validation outcomes are load-bearing and shape the UI:

1. **`TargetNotAllowedForKind`** — narrowing/changing `allowed_targets` to a target the *kind* doesn't support (e.g. `claude` on a `CodexAgent`). Already mapped to `library_target_not_allowed_for_kind` (422). The form must constrain its target checkboxes to the kind's legal set so this is a guard-rail, not a surprise.
2. **`TargetRemovedWithOverlays`** — dropping a target that has **overlay files** (Slice 5's `working/targets/<target>/`). Core refuses the drop unless `discard_orphan_overlays: true` is passed. This is **NOT yet mapped in `map_core_error`** (it falls to the `bridge_command_failed` catch-all / 502 today) — the one genuinely new error wiring this slice adds. It is the "destructive-adjacent" case the roadmap flagged (risk-a): the UI must surface it as a **confirm-then-discard** flow, never a silent strip and never an opaque 502.

This slice is **network-free and secrets-free** — it ships under the exact invariants Slices 3–5 proved. Unlike overlay writes (which don't commit), a metadata write **does commit** (the reference commits `metadata(<kind>/<name>)` — `commands.rs:770-778`), because `metadata.yaml` lives under git (it is NOT under the gitignored `working/`). That is the one way this slice's write posture differs from Slice 5 and matches Slice 4's publish/set-current commit-on-write.

## Decisions to settle (the roadmap flagged these; settled here)

### Decision 1 — The form edits exactly three fields: `allowed_targets`, `display_name`, `author`. `created_at` and `source_url` are read-only.

`MetadataUpdate` (core) carries only these three (plus the `discard_orphan_overlays` confirm flag). `created_at` is preserved verbatim by core (the UI never sends it). `source_url` is set-once at create-by-URL time and the reference's inspector never edits it (`metadata.rs:27`) — keep it read-only here (URL import is Slice 10b, post-Slice-8). The form is: a target checkbox group (constrained to the kind's matrix), a `display_name` text input (empty → `None`), an `author` text input (empty → `None`). Clearing a field to empty sends `null`, which core drops from the YAML (tested: `:281` display_name, `:782` author).

### Decision 2 — Metadata write COMMITS; the commit result rides back non-fatally, exactly like Slice 4's publish.

`metadata.yaml` is git-tracked (not under gitignored `working/`), so the reference commits it (`commands.rs:770`). This slice's write is therefore **snapshot-then-commit's metadata analogue**: core atomic-writes `metadata.yaml`, then `commit_change(&root, "metadata(<kind>/<name>)")` runs. **Reuse Slice 4's exact non-fatal posture:** a commit failure (no git identity, hook failure) is NOT fatal — the metadata write already landed — so it rides back as `{ committed, commit_error }` at the `ok` envelope level, never an error envelope (`main.rs:1035-1049` `commit_change` already returns `(bool, Option<String>)`). **But** `update_metadata` also returns the updated `PrimitiveMetadata` (core's return). **The envelope carries both:** `{ metadata: PrimitiveMetadata, committed: bool, commit_error: string|null }`. This is a small new result shape (call it `MetadataUpdateResult`) — mirror `PublishResult` but with the metadata payload attached. The UI uses `metadata` to refresh its render optimistically and surfaces `commit_error` (amber cue) if the commit failed but the write succeeded. **A non-git library** → `(committed: false, commit_error: null)` (the `.git`-absent silent skip, `commit_change:1036`) — the write still succeeded; no warning.

### Decision 3 — `TargetRemovedWithOverlays` is a two-phase confirm (409), NOT a silent strip and NOT a 422 dead-end.

(Roadmap risk-a.) When the user unchecks a target that has overlay files, the first save (with `discard_orphan_overlays: false`, the safe default — mirror `parse_force`) returns **409 `library_target_removed_with_overlays`**. The UI catches that code and shows a confirm: "Dropping <target> will delete its overlay file(s): <paths>. Discard them?" On confirm, it re-issues the **identical** PUT with `discard_orphan_overlays: true`, which deletes the orphaned overlay files in-core (`metadata.rs:96-101`) and proceeds. This is the **same captured-intent + re-issue-with-force pattern** as the install slice's `colliding_content` → `force:true` (D2) and Slice 5's destructive-adjacent overlay remove. **Open question O1:** the confirm copy wants to name the *specific* dropped targets + paths, which live in the error *payload* (`dropped: Vec<OverlayList>`), but `errorResult` forwards only `{code, message}`. Resolve in Phase 2 (Decision 2-of-Phase-2): either (a) the UI re-derives the affected overlays from the already-loaded `list_overlays` data (Slice 5 surfaces it — the UI knows which targets carry overlays without the payload), or (b) extend this one error to forward a structured `dropped` field. **Lean (a)** — the UI already has the overlay list from Slice 5's `listOverlays`, so it can name the paths client-side without weakening the never-forward-`detail` contract. The error payload stays server-side; the UI derives the confirm copy from data it already holds.

### Decision 4 — Target checkboxes are constrained to the KIND's matrix, so `TargetNotAllowedForKind` is a guard-rail, not a round-trip.

The form's checkbox group is driven by `kindInfo[detail.kind].allowed_targets` (the per-kind legal set, `api.ts:408`), not the full `Target::ALL`. A target the kind can't ship to is **never offered** as a checkbox. The 422 `library_target_not_allowed_for_kind` is then defense-in-depth (a malformed direct API call still 422s), never a path the form's own UI can reach. This mirrors Slice 5's "tabs driven by `allowed_targets`, never `Target::ALL`" discipline.
